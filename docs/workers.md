# Воркеры и очереди

Асинхронная работа — SQLite-based job queue. Worker — отдельный нативный процесс, запускается через `make up`.

## Где код

`backend/src/backend/infrastructure/queue/job_worker.py` — `JobWorker` класс, поллит SQLite-очередь.

## Текущие задачи

- `MEDIA` — вырезает скриншот и аудиофрагмент из видео для одного кандидата
- `MEANING` — генерация значений, переводов, синонимов через AI, батчем до 15 кандидатов
- `PRONUNCIATION` — скачивание произношения (US/UK)
- `VIDEO_DOWNLOAD` — скачивание YouTube-видео
- `TTS` — генерация TTS-аудио (делегируется в отдельный subprocess)

## TTS subprocess

TTS-задачи обрабатываются в отдельном subprocess (`backend/src/backend/infrastructure/queue/tts_subprocess.py`). Когда worker видит TTS-задачу, он spawn'ит subprocess, который загружает PyTorch/kokoro, обрабатывает все TTS-задачи из очереди и завершается. RAM от PyTorch полностью освобождается после завершения subprocess.

## Статусы фоновых операций

Статусы фоновых операций — через механизм джоб, а не через колонки в БД. Если операция выполняется асинхронно воркером — её прогресс отслеживается через статус джобы. Не добавлять `*_status`-колонки для отслеживания прогресса фоновых задач.

## Принцип обработки задачи

Каждая задача сначала помечает свои объекты как `RUNNING` в отдельной commit'нутой сессии (чтобы прогресс был виден в UI даже при откате основной), затем выполняет use case. Permanent-ошибки (`PermanentAIError`, `PermanentMediaError`) сразу ведут к `FAILED` без ретраев; на остальных — ретрай с финальной маркировкой `FAILED` на последней попытке. Детали реализации — в `job_worker.py`.

## Команды

Логи воркера смотрим **только через `make`** — общим потоком со всеми остальными сервисами (детали — `docs/running.md`):

```
make logs         # app + worker + ai_proxy (текущая рабочая копия)
```

Если когда-нибудь понадобится смотреть логи только воркера — это повод **добавить `make`-таргет**. Все логи должны быть доступны через `make`.
